What's Wrong & r2

May 9, 2026

The microservices vs monolith debate is one of software's most recurring arguments. Both camps are wrong; not about the tradeoffs, but about what causes them. The forced choice is a language-level failure.

No mainstream language lets you say: evaluate this expression on a different node, with redundancy, and flow the result back transparently. So we use microservices as a blunt instrument — deployment boundary as a substitute for a language feature. And when that's too painful, we give up and build a monolith. Neither side is asking why the choice exists at all.

Microservices give you:

  • Forced interface discipline between components
  • Independent scaling per service
  • Redundancy at the service level across availability zones

Monoliths give you:

  • Coupling you can actually reason about
  • One deployment artifact, far simpler operationally
  • Refactoring across components without a network contract

Rust's traits and Go's interfaces are getting this partially right at the type-system level. Decoupling interface from implementation without carving everything into network-connected processes. But neither gives you distributed evaluation. The real fix is a language where "run this on three nodes and return the first result" is a library call, not an infrastructure project. Something like Plan 9's networked philosophy (in a perfect world where it was good). Unison is the closest existing attempt but its type system is too static.

DevOps is an abstraction problem

Build systems, package managers, init systems, and cluster orchestrators all manage the same thing: dependency graphs, execution order, caching, and side effects. We (well, not me) built separate, incompatible, broken tools for each domain.

Model the problem correctly: Excel, React, Haskell (Read Build Systems a la Carte, friend)

Don't hand-roll shell scripts or use Dockerfiles.

Excel, React, and Haskell model computation as a DAG with explicit data flow. Same inputs produce same outputs. Caching is principled. Make's dependency model is fragile. Shell scripts have no dependency model — they're sequences of mutations. Dockerfiles are worse: they represent a build as a linked list of snapshots. The actual structure is a DAG. Are we serious with this linked list situation? This is why invalidating one layer ruins everything downstream and why hermetic builds are so hard to achieve in practice.

Terraform is duct tape — HCL over infrastructure APIs that have no principled model. The various tools in the ecosystem like Kubernetes, Ansible, Vagrant, Sentinel, Jenkins blah blah all evaporate if the correct substrate exists.

Nix is gooderer: packages as pure functions of their inputs, the whole system as a derivation graph. But Nix requires build graphs to be fully serialized before evaluation, which is a real limitation. And at runtime, NixOS behaves nearly identically to any other Linux distribution — it's Nix all the way to the shell, then POSIX the rest of the way down.

The thunk solution

Building LLVM has ~50,000 compilation units. Enumerating the entire build graph upfront before any evaluation is expensive. You want to represent LLVM as a single node that, when forced, expands into 50,000 nodes. That's a thunk. Nix handles this awkwardly via "dynamic derivations" and a socket mechanism. A lazily-evaluated language gets it for free — the graph expands on demand, evaluation handles it, no special-casing needed.

Static is a perspective, not a foundation

Data is dynamic. Hardware is dynamic. Storage degrades, peripherals come and go, topology changes. Static subsets exist within dynamic systems. But the top level is always dynamic; staticness is relative to a frame of reference, not a global property.

Managing dynamic systems with static languages requires emulating a dynamic language inside them (see r2!). That emulation is always more cumbersome and more error-prone than just having a dynamic language. PS/2 was a static interface — one rigid protocol, no hotplug so USB replaced it.

The ideal system is a dynamic, lazy, pure-by-default language that is the OS — not a language running on an OS. Everything is an expression: users, services, the filesystem. Process management falls out of lazy evaluation semantics. Capabilities replace Unix's ambient authority model: every expression is passed only what it needs, can request more from its parent, and resource limits (CPU, memory) are first-class. A cross-node semaphore with evenly distributed entries should be a few lines of normal code, not a multi-week integration project.

Self, Plan 9, and the best parts of Nix each had the right instinct in one dimension. None got all the way there. Kubernetes, Terraform, Dockerfiles, Make, and shell scripts all exist because the OS and language layer below them failed to model the actual problem. The fix isn't another abstraction layer. It's going back to the foundation.

r2: a first attempt

r2 is a Rust runtime for content-addressed lazy computation with explicit object-capability effects. It's not a programming language, not a Kubernetes replacement, not production-ready. It's the first layer — a proof that the substrate can be built correctly.

The thesis: build systems, package managers, init systems, deployment tools, and cluster orchestrators are fragmented solutions to the same problem. r2 tries to unify them under one runtime model.

How it works

Everything is a graph node. Every node has a stable SHA-256 hash. Evaluation happens by forcing nodes lazily — you ask for the result of a node, and only the necessary subgraph gets evaluated.

  • Value: Literal: Int, Text, Bytes, Tree, Artifact
  • Thunk: Lazy indirection to another node
  • Apply: Pure function application over graph nodes
  • HostCall: Effectful call, gated by a registered capability
  • Action: Hermetic external command with declared inputs/outputs
  • ReadCell: Reads a mutable cell; the runtime's escape hatch for live state

Effects are typed. Every node knows whether it's pure, hermetic, or live, and the runtime enforces consistency — using a Live capability in a pure context produces a structured EffectMismatch failure.

  • Pure: Deterministic, no outside world access. Always cacheable.
  • Hermetic: Effectful but deterministic given declared inputs. Cacheable by action hash.
  • Live: Touches the open world — network, clock. Not permanently cacheable.

No ambient authority. The runtime only knows host functions explicitly registered via CapSet — no global filesystem, network, PATH, or shell by default. Missing capabilities produce structured UnknownCapability failures. This isn't a nice-to-have; it's the enforcement mechanism that makes caching trustworthy.

Cells provide the mutable-state escape hatch for live services and temporal data. A ReadCell node records the cell version it read; when the cell updates, downstream nodes become stale. It's the same model as a spreadsheet — controlled mutation that propagates invalidation through the graph.

In code

let mut rt = Runtime::new(".r2")?;

rt.register("+", HostFn::pure(|args| match args {
    [Value::Int(a), Value::Int(b)] => Ok(Value::Int(a + b)),
    _ => Err(FailureKind::TypeError("+ expects two ints".into())),
}));

let a = rt.int(20)?;
let b = rt.int(22)?;
let sum = rt.thunk(rt.call("+", vec![a, b])?)?;

let first  = rt.force(sum.clone())?;  // cache_hit: false
let second = rt.force(sum)?;           // cache_hit: true

Hermetic builds work the same way — import a source tree, declare tool inputs and output paths, force the action node. Outputs are content-addressed and cached. Running the same compilation twice hits the cache. Changing one source file invalidates only the affected subgraph.

let src    = rt.import_tree("./src")?;
let cc     = rt.import_tool("/bin/cc")?;
let action = rt.action(ActionSpec {
    program: "/bin/cc".into(),
    tool: cc,
    inputs:  vec![ActionInput { path: "src".into(), hash: src }],
    outputs: vec!["main.o".into()],
    ..Default::default()
})?;

let result = rt.force(action)?;
